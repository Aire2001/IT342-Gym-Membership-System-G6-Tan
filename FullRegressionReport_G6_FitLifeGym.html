2. Refactoring Summary

The project was refactored from a traditional layered architecture (controllers/, services/, dto/, entity/, repository/, exception/ at the root level) to a Vertical Slice Architecture (VSA) where all code for a feature — controller, service, DTOs, and domain models — is co-located inside a single feature folder, with shared infrastructure extracted into a shared/ package.

2.1 Objectives of the Refactoring

  • Organize the codebase by feature/module rather than by technical layer
  • Ensure all existing features remain functional after restructuring
  • Maintain clean code and proper naming conventions throughout
  • Improve maintainability and separation of concerns per feature slice
  • Extract shared infrastructure (entities, repositories, config) into a dedicated shared/ package

7. Issues Found

During the Vertical Slice Architecture refactoring, five technical issues were identified and resolved before the final test run. All issues were regressions introduced during the file reorganization process, not pre-existing bugs in the business logic. No functional regressions were detected after fixes were applied.

8. Fixes Applied

Fix 1 — Created missing RegisterRequest.java

Created feature/auth/RegisterRequest.java with fields firstname, lastname, email, password, confirmPassword and Jakarta validation annotations (@NotBlank, @Email, @Size). This matches the contract expected by AuthService.register() and AuthController.

Fix 2 — Stripped UTF-8 BOM from 40 Java source files

Used PowerShell to read each .java file with [System.IO.File]::ReadAllText(), strip the leading U+FEFF character using TrimStart(), then re-write with UTF8Encoding(false) (no BOM). All 40 files were corrected in one pass.

Fix 3 — Added missing import to LoginResponse.java

Added import edu.cit.tan.GymMembershipPaymentSystem.shared.dto.UserResponse; to feature/auth/LoginResponse.java. The class was moved during refactoring but the import was not updated to reflect the new shared.dto package location.

Fix 4 — GlobalExceptionHandler: proper HTTP status for AccessDeniedException

Added explicit handler methods for AccessDeniedException (returns 403) and AuthenticationException (returns 401) in GlobalExceptionHandler.java. Previously, the catch-all @ExceptionHandler(Exception.class) was intercepting Spring Security exceptions and wrapping them as HTTP 500 responses, causing TC-MEM-004 and TC-MEM-006 to fail.

@ExceptionHandler(AccessDeniedException.class) public ResponseEntity<Map<String, String>> handleAccessDenied(AccessDeniedException ex) { Map<String, String> response = new HashMap<>(); response.put("error", "Access denied: " + ex.getMessage()); return new ResponseEntity<>(response, HttpStatus.FORBIDDEN); }

Fix 5 — TypeScript null-check in AdminPanelPage

Changed editingRole.role → editingRole?.role ?? '' and editingStatus.status → editingStatus?.status ?? '' in features/admin/AdminPanelPage.tsx. TypeScript strict mode requires explicit null guards even when values are accessed inside optional-chained conditional expressions. This unblocked the npm run build production build.
