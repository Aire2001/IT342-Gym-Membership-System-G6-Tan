1. Executive Summary

1.1 Project Overview

The FitLife Gym Membership and Payment System is a multi-platform application developed for IT342 — System Integration and Architecture. It manages gym members, membership plans, and payment transactions across three platforms: a Spring Boot backend REST API, a React web frontend, and an Android mobile application. Members can register, log in, view their membership status, and pay fees online. Administrators can manage memberships and monitor all payments from a dedicated admin panel.

This document serves as both the system design documentation and the full regression test report for the IT342 Phase 4 submission, covering the complete refactoring to Vertical Slice Architecture and the automated testing that verified all existing features remained functional afterward.

1.2 Objectives

  • Provide a centralized system for gym members to manage their memberships online
  • Integrate an online payment system supporting GCash, Credit Card, and Bank Transfer
  • Allow gym administrators to monitor payments and manage plans efficiently
  • Eliminate manual tracking of payments and member records
  • Refactor the entire codebase to Vertical Slice Architecture for better maintainability
  • Verify through automated testing that all features remain intact after refactoring

1.3 Scope

Features Included

  • User registration and login with JWT authentication
  • Membership plan management — view, add, update, and delete plans
  • Online payment processing with GCash, Credit Card, and Bank Transfer
  • Member dashboard showing membership status and payment summary
  • Payment history with downloadable receipts and search and filter
  • Membership status indicator showing Active, Expired, or Pending
  • User profile management and logout
  • Admin panel for managing users, plans, and all payment records

Features Excluded

  • Fitness progress tracking, class scheduling, trainer booking
  • Push notifications and email reminders
  • Workout plans, diet or nutrition tracking, wearable device integration

3. Functional Requirements Specification

3.1 Core User Journeys

Journey 1 — Gym Member Registration and Login

  1. User opens the system on the web or mobile application
  2. User fills in the registration form with their first name, last name, email, and password
  3. The system validates the input, creates the account, and automatically logs the user in with a JWT token
  4. User is redirected to the dashboard showing their membership status and recent payments

Journey 2 — Membership Payment

  1. Logged-in user views their current membership status on the dashboard
  2. User navigates to the membership plans page and selects a plan
  3. User chooses a payment method — GCash, Credit Card, or Bank Transfer — and confirms
  4. GCash and Credit Card payments complete instantly; Bank Transfer is set to Pending until manually verified
  5. Membership status updates and the user can download a payment receipt

Journey 3 — Administrator Membership Management

  1. Admin logs in with admin credentials and is directed to the admin panel
  2. Admin can view all users, all payments, and all membership plans
  3. Admin can add new membership plans, update existing ones, or delete plans no longer offered
  4. Admin can update payment status for Bank Transfer transactions after verifying receipt

5.2 Component Overview

The system follows a three-tier client-server architecture. Both the React web application and the Android mobile application act as clients that communicate with the Spring Boot backend through HTTP REST calls. The backend handles all authentication, business logic, and data persistence. The MySQL database is only accessible through the backend — frontend clients never connect to it directly.

Every request from a client includes a Bearer JWT token in the Authorization header. The backend's token filter validates the token and loads the user and their role before the request reaches the controller. Unauthenticated requests to protected endpoints are rejected with HTTP 401 and insufficient-permission requests are rejected with HTTP 403.

9. Refactoring Summary — Vertical Slice Architecture

9.1 What Changed and Why

The entire codebase was refactored from a traditional layered architecture to a Vertical Slice Architecture (VSA) as required by IT342 Phase 4. In the original structure, all controllers were in one folder, all services in another, all DTOs in another, and so on. A developer working on a single feature had to jump across multiple folders to find all the relevant code.

In Vertical Slice Architecture, everything related to one feature — its controller, service, and request/response objects — lives together in a single feature folder. Code shared across features (database entities, repositories, shared DTOs, security configuration, exception handling) is placed in a dedicated shared package. This makes each feature self-contained and far easier to understand, modify, and test independently.

9.2 Changes Applied by Platform

PlatformBefore — Layered ArchitectureAfter — Vertical Slice Architecture
Backend (Spring Boot) All controllers together, all services together, all DTOs together, all entities together, all repositories together, exception handlers together One folder per feature: auth, membership, payment, dashboard, profile, admin, user, home — each with its own controller, service, and request/response classes. Shared entities, repositories, and exception handler in a shared package.
Web Frontend (React) All pages in a pages folder, all reusable components in components folder, all API calls in a services folder, authentication context in context folder, TypeScript types in types folder One folder per feature: auth, dashboard, membership, payment, admin, profile — each with its page component and its own API file. Shared navigation component, TypeScript types, and Axios client in a shared folder.
Mobile (Android Kotlin) All screens and all network classes mixed in the root package without feature organization One folder per feature: auth, dashboard, membership, payment — each with its screen file. Shared Retrofit client, API service interface, and data models in a shared network folder.

9.3 Objectives Achieved

  • All features are organized by business domain rather than by technical layer across all three platforms
  • All existing features remain fully functional — confirmed by 20 automated tests all passing
  • Naming conventions are consistent and follow the original project's conventions throughout
  • Shared infrastructure is cleanly separated from feature-specific code
  • The codebase is more maintainable — all code for any feature can now be found in one place

2 Hidden Encoding Marker on All 40 Java Source Files — Backend
When the Java source files were generated using PowerShell during the refactoring, a hidden byte-order mark character was automatically placed at the very beginning of every file. The Java compiler does not recognize this character as valid source code and rejected all 40 files with an illegal character error, making it impossible to compile the backend.		 Backend — All 40 Java Source Files Critical — compile error A script was written to open each of the 40 Java source files, detect and remove the hidden marker character from the beginning of the file, and re-save each file in standard UTF-8 encoding without any marker. All 40 files were corrected in a single automated pass without changing any of the actual source code inside them. Fixed
3 Missing Package Reference in Login Response — Backend
The UserResponse class was moved to a new shared package during refactoring, but the login response file was never updated to point to the new location. The compiler could not find the class when building the auth feature and the project failed to compile.		 Backend — Auth Feature — Login Response File High — compile error The login response file was updated to include the correct package path pointing to the shared data transfer folder where UserResponse was relocated during the refactoring. This single-line correction resolved the compile error and the project built successfully. Fixed
4 Security Errors Returning Wrong HTTP Status — Backend
The global error handler had a general catch-all that intercepted Spring Security's access-denied errors before the dedicated security handler could process them. This caused those errors to be returned as HTTP 500 Internal Server Error instead of the correct HTTP 403 Forbidden. Two membership tests were failing because they expected 403 but received 500.		 Backend — Shared Exception Handler Medium — wrong HTTP status, tests failing Two dedicated error handler methods were added to the global exception handler — one for access-denied errors that returns HTTP 403 Forbidden, and one for authentication failures that returns HTTP 401 Unauthorized. These dedicated handlers take priority over the general catch-all, so Spring Security errors are now always processed with the correct status code. Tests TC-MEM-004 and TC-MEM-006 both passed after this fix. Fixed
5 TypeScript Null-Safety Errors in Admin Panel — Web Frontend
Two property accesses in the admin panel page were flagged by TypeScript's strict mode as potentially unsafe. The properties were accessed on state objects that TypeScript considered possibly null inside optional-chained conditional expressions. TypeScript strict mode does not automatically narrow types from this pattern and blocked the entire frontend production build.		 Web Frontend — Admin Feature — Admin Panel Page Medium — frontend build failure The two property accesses were updated to use null-safe access combined with a fallback empty string value. When the editing state objects are absent, an empty string is returned in their place rather than attempting to access a property on a null value. This satisfied TypeScript's strict null-checking requirements and the production frontend build completed successfully. Fixed
